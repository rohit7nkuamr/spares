Project Sentinel: Unified AI Intelligence Platform

This project transforms the Indian Army’s maintenance and logistics to a proactive, data-driven framework. **Target Release Date: 13-01-2026.**

Project Status & Experience

  • Status: Partially developed. Ready for full integration.
  • Proven Pilot: Similar mobile/desktop app successfully delivered and tested at **474 EME wksp, Zirakpur**.
  • Indigenous Content: 100% Indigenous program/app development.

Critical Challenges Addressed

  • Unscheduled Downtime: Predicting failures before they compromise operations.
  • Inefficient Resource Allocation: Shifting from fixed schedules to condition-based maintenance.
  • Suboptimal Spares: Mitigating critical shortages and overstocking simultaneously.

Strategic Impact

Sentinel ensures maximum fleet readiness, enhances operational availability, and optimizes the entire logistical support chain. This aligns with the national strategic imperative of using advanced technology (Atma Nirbhar Bharat).

Expected Impact

Quantitative Gains

The primary goal is a 20-30% increase in equipment uptime and significant reductions in inventory costs. This chart shows the projected average improvement.

Provides AI-driven equipment ratings and forecasts of maintenance interventions during operations.

Self-Reliance

Role-Based Access & User Scenarios

The application is built around the specific information needs of three key operational personas. Click a card to view their core goals and a key scenario.

Jawan (Technician) | EME Corps

Focus: Hands-on maintenance, field operations (Mobile User).

Core Goal: Receive clear, specific work orders (e.g., "Check ALS Cooling System").

Scenario: Jawan Singh at 474 EME Wksp.

Receives **mobile ALERT**: "ALS Vehicle BA-01-A0123 - High Probability of Cooling System Failure (85%)." The app shows specific data (temp, pressure) that triggered the prediction. He uses the mobile checklist to log the repair and part used (faulty hose).

JCO (Workshop Commander)

Focus: Task prioritization, resource management (Desktop/Web User).

Core Goal: Single pane of glass view to prioritize maintenance based on asset criticality.

Scenario: JCO Verma on Desktop Dashboard.

Sees the workshop's fleet health: 40 Green, 7 Amber, 3 Red (Critical). He checks the logistics module, confirms the required critical spare is in transit, and assigns his best technician to the task via the app.

Commissioned Officer | Logistics & Ordnance

Focus: Demand forecasting, supply chain optimization (Desktop/Web User).

Core Goal: Accurately forecast demand for thousands of spare parts for the next quarter.

Scenario: Major Gupta preparing Forecast.

Accesses the SDF module. The dashboard provides a 6-month forecast for critical spares, augmented by predicted engine failures due to operational intensity. He uses this data to submit a highly accurate, risk-mitigated indent.

Implementation Roadmap (6-Month Prototype Cycle)

The project follows a 6-month phased approach to deliver a tested and validated pilot system, starting with the pilot vehicle type, the **Ashok Leyland Stallion (ALS)**.

Phase 1: Foundation & Scoping (Weeks 1-4)

  • Kickoff: Finalize PRD/Tech Spec and establish CI/CD Pipeline.
  • Data & Schema: Implement Core Database Schema (PostgreSQL/Time-Series DB).
  • Key Milestone: Joint Data Scoping Workshop with Army SME (Week 2).
  • Deliverable: Functional Development & Staging Environments.

Phase 2: Core Feature Development (Weeks 5-16)

  • Backend: Implement User Authentication (RBAC) and Equipment/Log APIs.
  • Frontend: Build Fleet Health Dashboard, Asset Views, Alerting UI, and Spares Forecast Dashboard.
  • AI/ML: **Develop, Train & Validate v1.0 AI Models for ALS** (Failure Prediction & Spares Demand).
  • Key Milestone: Internal Alpha Release – Full integration of all services and v1.0 AI models (Week 16).

Phase 3: Integration, Testing & Handover (Weeks 17-24)

  • Testing: Conduct comprehensive System Integration (SIT) and Security/Load Testing.
  • UAT & Training: **User Acceptance Testing (UAT)** with designated personnel from **474 EME Wksp** (Weeks 17-20).
  • Required Assistance: Ord/EME SME required for feature engineering and process validation (4+ hours/week).
  • Key Milestone: Production Deployment, Formal User Training, and Documentation Handover (Week 24).

Critical Dependencies and Risk Mitigation

Success hinges on collaboration. These are the key dependencies, risks, and the agreed-upon strategies to ensure project delivery is not compromised.

Dependency 1: Data Quality & Access

The AI model accuracy depends entirely on the completeness and consistency of historical maintenance logs.

Mitigation Strategy:
  • Mandatory joint data-scoping workshop (Week 2).
  • Co-create explicit Data Requirements Document (DRD).
  • Build automated data validation scripts into the ingestion pipeline.

Dependency 2: Consistent Army SME Access

Expert EME/Ordnance guidance is critical for feature engineering and practical validation of maintenance triggers.

Mitigation Strategy:
  • Formally assign a Primary and Secondary SME.
  • Establish a fixed schedule of interactions (min. 4 hours/week).
  • Utilize asynchronous tools to respect SME operational duties.

Dependency 3: UAT/Pilot Unit Availability

Availability of personnel and equipment at 474 EME Wksp is critical for the UAT and training window (Weeks 17-20).

Mitigation Strategy:
  • Schedule UAT/training periods 2+ months in advance with Command.
  • Develop detailed UAT plan for maximum time efficiency.
  • Build a one-week buffer into the project plan.

Hybrid Database & Data Ingestion

The **Hybrid Database Strategy** is a critical design choice, separating low-volume transactional data from high-volume sensor readings to ensure long-term performance and scalability.

1. Data Types & Storage

  • **Relational Data (PostgreSQL):** Used for structured, transactional data like **Equipment Master Registry, User Accounts, and Maintenance Logs**.
  • **Time-Series Data (InfluxDB/TimescaleDB):** Essential for high-volume, high-velocity time-stamped data from **Future IoT Sensors** (temp, vibration, pressure).

2. Ingestion Methods

  • **Historical Data:** Bulk uploads via secure web interface (CSV/Excel).
  • **Manual Data:** Technicians log maintenance/spares usage directly via the Mobile App.
  • **Future IoT Data:** Secure, scalable **MQTT Broker** and REST API endpoints for real-time sensor streams.

MLOps (Model Lifecycle Management)

MLOps is a foundational, non-negotiable component for managing the **dozens of independent models** required for diverse vehicle fleets (e.g., ALS Engine vs. T-90 Transmission).

1. Core MLOps Components (MLflow)

  • **Experiment Tracking:** Log all training runs, parameters, and metrics.
  • **Model Registry:** Central repository to store, version, and stage trained models (Staging to Production).
  • **Automated Pipelines (CI/CD):** Triggers model retraining when new data is available or performance degrades (Model Drift).

2. Model Deployment & Monitoring

  • **Serving:** Models are containerized and deployed as microservices with a REST API endpoint for inference.
  • **Monitoring:** Dedicated dashboard tracks real-world model performance (Accuracy, Precision, Recall) to ensure models remain relevant.
